Section 1

The big picture: what are we even building?

Imagine you rent space in a giant shared building (that's AWS). You don't want strangers wandering through your rooms, so the first thing you do is put up walls and lock the doors. In AWS, those walls are called a VPC — a Virtual Private Cloud. It is your own private, fenced-off section of Amazon's cloud where only the things you allow can talk to each other.

Inside that fenced area, you divide the space into rooms. Some rooms have a door to the street (the public internet) — we call those public subnets. Other rooms have no street door at all; you can only reach them from inside the building — those are private subnets. Sensitive things like your databases live in the private rooms. The friendly front desk that greets visitors lives in a public room.

A house analogy you can keep in your head

  • VPC = the whole house and its fence.
  • Subnet = a room in the house.
  • CIDR block = how many "addresses" (spots for furniture/people) a room can hold.
  • Internet Gateway = the front door to the street.
  • NAT Gateway = a one-way mail slot: people inside can send letters out, but no one outside can walk in.
  • Route table = the signs that tell traffic which door to use.
  • Security Group / NACL = the locks and the guard checking who's allowed through each door.

Section 2

Networking 101: IP addresses, in plain words

Every device on a network needs an IP address — a number that works like a street address so messages know where to go. The common form (IPv4) looks like 10.0.5.23: four numbers from 0–255, separated by dots. That gives about 4.3 billion possible addresses worldwide — which sounds like a lot until the whole planet is online, which is why some addresses are set aside as private.

Private addresses are reusable inside your own network and never travel on the open internet. A standards document called RFC 1918 reserves three private ranges that everyone uses for internal networks:

Private rangeSizeTypical use
10.0.0.0/8~16.7 million addressesBig company networks & most AWS VPCs
172.16.0.0/12~1 million addressesMedium networks (avoid 172.17.0.0/16 on AWS — see note)
192.168.0.0/16~65,000 addressesHome routers & small networks
⚠ AWS gotcha: avoid 172.17.0.0/16 for your VPC. Some AWS services (Cloud9, SageMaker) and Docker use it internally, and overlapping it causes hard-to-debug connection failures. AWS docs ↗

We use 10.0.0.0/16 for our VPC because the 10.x range is huge, leaves the most room to grow, and is the convention almost every AWS team follows. The /16 part is the magic — that's CIDR, and it's next.

Section 3

CIDR: the /16, /24, /26 mystery solved

CIDR stands for Classless Inter-Domain Routing. Forget the scary name — it's just a way to say "how big is this network?" The number after the slash tells you how many bits of the address are fixed (the network part). The remaining bits are free to count up through individual addresses.

An IPv4 address has 32 bits total. So:

/16 → 32 − 16 = 16 free bits → 216 = 65,536 addresses
/20 → 32 − 20 = 12 free bits → 212 = 4,096 addresses
/22 → 32 − 22 = 10 free bits → 210 = 1,024 addresses
/24 → 32 − 24 = 8 free bits → 28 = 256 addresses
/26 → 32 − 26 = 6 free bits → 26 = 64 addresses
/28 → 32 − 28 = 4 free bits → 24 = 16 addresses (smallest AWS allows)

The rule of thumb: a smaller number after the slash means a bigger network. /16 is a mansion; /28 is a closet. AWS lets a VPC be anywhere from /16 (biggest) to /28 (smallest).

The "5 reserved IPs" tax — don't forget it

In every subnet, AWS quietly keeps 5 addresses for itself, so you never get the full count:

AddressReserved for
.0 (first)Network address
.1The VPC router (the default gateway)
.2AWS DNS server (base of range + 2)
.3Reserved by AWS for future use
.255 (last)Network broadcast (AWS doesn't support broadcast, but reserves it)

So a /24 (256 addresses) really gives you 251 usable. A /28 (16) gives only 11. This is exactly why you don't make production subnets too tiny. Subnet sizing docs ↗

Section 4

Subnets & the three private sizes

A subnet ("sub-network") is a slice of your VPC's address range that lives in exactly one Availability Zone (AZ) — a physically separate datacenter. We repeat the same set of subnets in three AZs so that if one datacenter has a problem, the other two keep serving. That's the foundation of high availability, and it costs nothing extra to plan for.

Key idea most beginners miss: AWS does not label a subnet "public" or "private." A subnet is public only because its route table sends internet-bound traffic (0.0.0.0/0) to an Internet Gateway. Remove that route and the exact same subnet becomes private. Routing decides behavior — not the name, not the size.

Why three private sizes instead of one?

Different workloads need wildly different numbers of IP addresses. A tiny helper service might need 10 IPs forever; an OpenSearch or Kafka cluster that auto-scales can chew through hundreds. If you give everything the same size you either waste a huge range on small things or starve the big things. So we cut three right-sized tiers and stamp each one into all three AZs:

TierPer-AZ CIDRTotal / Usable IPsWhat lives here & why that size
Small /26 64 / ~59 Low-IP services: VPC interface endpoints for SNS/SQS, small internal microservices, admin/utility pods. These rarely scale out, so a closet is plenty and you avoid wasting address space.
Medium /24 256 / ~251 The default home for most compute: Java & Node.js web apps, backend APIs, NiFi dataflow nodes. Enough room for rolling deploys (old + new running together) and moderate auto-scaling.
Large /22 1,024 / ~1,019 IP-hungry clusters: Kafka (MSK) brokers, Postgres/RDS fleets, OpenSearch data nodes. These add nodes as data grows and each node + its cross-AZ replicas eats IPs fast. Roomy on purpose.

Notice the deliberate gaps in the plan below. Each tier sits on its own /20 "lane," and a big /18 is left completely empty. Leaving gaps is a best practice: you cannot resize a subnet after creation, so reserving room now means you can add a fourth tier later without renumbering everything.

VPC 10.0.0.0/16 (65,536 addresses)
├─ public 10.0.0.0/20 → /24 per AZ
├─ private-small 10.0.16.0/20 → /26 per AZ
├─ private-medium 10.0.32.0/20 → /24 per AZ
├─ private-large 10.0.48.0/20 → /22 per AZ
└─ RESERVED 10.0.64.0/18 → (left empty for the future)

Best practices (do)

  • ✓ Use one /16 VPC and carve predictable tiers.
  • ✓ Mirror subnets across ≥2 (ideally 3) AZs.
  • ✓ Right-size each tier; leave numbering gaps.
  • ✓ Keep databases & brokers in private subnets only.
  • ✓ Put only load balancers + NAT in public subnets.

Anti-patterns (avoid)

  • ✗ One giant flat subnet for everything.
  • ✗ Tiny /28 subnets for things that scale.
  • ✗ Overlapping CIDRs (breaks future VPC peering).
  • ✗ Putting databases in public subnets.
  • ✗ Allocating the whole /16 with no room left.

Section 5

Security in layers (internal & external)

Good cloud security is like an onion — many layers, so that if one fails, another still protects you. This is called defense in depth. AWS gives you several independent controls; we use them together rather than relying on any single one.

Layer 1 — Security Groups (stateful, per-resource)

A Security Group is a firewall wrapped around a specific resource (an app server, a database). "Stateful" means if you allow a request in, the reply is automatically allowed back out — you don't write a return rule. The best-practice trick: rules reference other security groups, not IP ranges. Our database SG says "allow Postgres from the app security group," so it doesn't matter what IPs the app servers get — only members of the app tier can connect. This is internal segmentation between tiers.

Layer 2 — Network ACLs (stateless, per-subnet)

A NACL is a coarser firewall at the subnet door. "Stateless" means it checks every packet in both directions independently — you must explicitly allow return traffic on high-numbered "ephemeral" ports. We use a NACL as a backstop on private subnets. Security Groups are your primary tool; the NACL is the extra layer in case a SG is misconfigured.

Layer 3 — Public vs private placement

The strongest control is simply not being reachable. Databases, Kafka, and OpenSearch sit in private subnets with no route to the internet. The only thing exposed externally is a load balancer on port 443 in the public subnet. Everything else is internal-only.

Layer 4 — VPC Endpoints (private paths to AWS services)

Normally, calling SQS, SNS, or S3 sends traffic out to the public AWS endpoints. VPC Endpoints create a private doorway directly from your subnets to those services, so messaging traffic never touches the public internet. We add a free Gateway endpoint for S3 and Interface endpoints for SQS and SNS.

Layer 5 — Flow Logs (visibility & audit)

You can't secure what you can't see. VPC Flow Logs record every accepted and rejected connection to CloudWatch, giving you an audit trail for investigations and a way to spot unexpected traffic.

Section 6

The Terraform code, file by file

Terraform lets you describe infrastructure as text files. You write what you want; Terraform figures out the API calls to make it real, and remembers state so it only changes what's needed. We split the project into focused files — the network, the security layer, the self-hosted data services, and the public app edge — a common best practice that keeps each concern readable. The download button at the very bottom gives you all of these as ready-to-run files.

The whole workflow is three commands: terraform init (download the provider) → terraform plan (preview changes) → terraform apply (make them real). Always read the plan before you apply. The base network applies on its own; flip enable_example_workloads = true to add the data services and app edge.

main.tf

The network itself: the VPC, the public plus three private subnet tiers, the internet gateway, NAT gateways, and one route table per AZ.

# main.tf
# ---------------------------------------------------------------------------
# CIDR PLAN (VPC 10.0.0.0/16 = 65,536 addresses)
# Each subnet loses 5 usable IPs to AWS (.0 network, .1 router, .2 DNS,
# .3 future, and the last broadcast address).
#
# We carve the /16 into named "tiers" by the SECOND octet group so the plan is
# readable at a glance and leaves big gaps for future expansion.
#
#   PUBLIC          10.0.0.0/20   -> /24 per AZ  (256 IPs each, ~251 usable)
#   PRIVATE-SMALL   10.0.16.0/20  -> /26 per AZ  (64 IPs,  ~59 usable)  small svcs
#   PRIVATE-MEDIUM  10.0.32.0/20  -> /24 per AZ  (256 IPs, ~251 usable) general
#   PRIVATE-LARGE   10.0.48.0/20  -> /22 per AZ  (1024 IPs,~1019 usable) big data
#   RESERVED        10.0.64.0/18  -> untouched, room to grow / future tiers
#
# WHY THREE PRIVATE SIZES?
#   small  -> low-IP-count workloads: SNS/SQS access points, small APIs,
#             bastion-free admin endpoints, lightweight microservices.
#   medium -> the default home for most app servers: Java/Node web apps,
#             backend APIs, NiFi nodes.
#   large  -> IP-hungry clustered systems: Kafka (MSK) brokers, RDS/Postgres
#             fleets, OpenSearch data nodes, search clusters that scale out.
# ---------------------------------------------------------------------------

# Look up which AZs actually exist in the chosen region (don't hard-code).
data "aws_availability_zones" "available" {
  state = "available"
}

locals {
  azs = slice(data.aws_availability_zones.available.names, 0, var.az_count)

  # cidrsubnet(base, newbits, netnum) splits a block deterministically.
  # Public: 4 newbits on a /20 base -> /24 subnets.
  public_subnets = {
    for i, az in local.azs : az => cidrsubnet("10.0.0.0/20", 4, i)
  }

  # Small: 6 newbits on /20 -> /26 subnets (64 IPs).
  private_small_subnets = {
    for i, az in local.azs : az => cidrsubnet("10.0.16.0/20", 6, i)
  }

  # Medium: 4 newbits on /20 -> /24 subnets (256 IPs).
  private_medium_subnets = {
    for i, az in local.azs : az => cidrsubnet("10.0.32.0/20", 4, i)
  }

  # Large: 2 newbits on /20 -> /22 subnets (1024 IPs).
  private_large_subnets = {
    for i, az in local.azs : az => cidrsubnet("10.0.48.0/20", 2, i)
  }

  name_prefix = "${var.project_name}-${var.environment}"
}

# ---------------------------------------------------------------------------
# NAT GATEWAYS — let private subnets make OUTBOUND connections (patch
# downloads, calling external APIs) without being reachable from the internet.
# Each NAT needs a public Elastic IP and lives in a PUBLIC subnet.
# ---------------------------------------------------------------------------
locals {
  nat_azs = var.single_nat_gateway ? slice(local.azs, 0, 1) : local.azs
}

resource "aws_eip" "nat" {
  for_each = toset(local.nat_azs)
  domain   = "vpc"
  tags     = { Name = "${local.name_prefix}-nat-eip-${each.key}" }
}

resource "aws_nat_gateway" "this" {
  for_each      = toset(local.nat_azs)
  allocation_id = aws_eip.nat[each.key].id
  subnet_id     = aws_subnet.public[each.key].id
  tags          = { Name = "${local.name_prefix}-nat-${each.key}" }

  depends_on = [aws_internet_gateway.this]
}

# ---------------------------------------------------------------------------
# ROUTE TABLES
# A subnet is "public" only because its route table sends 0.0.0.0/0 to the IGW.
# Private subnets send 0.0.0.0/0 to a NAT Gateway instead.
# ---------------------------------------------------------------------------

# Public route table (shared by all public subnets).
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.this.id
  tags   = { Name = "${local.name_prefix}-rt-public" }
}

resource "aws_route" "public_internet" {
  route_table_id         = aws_route_table.public.id
  destination_cidr_block = "0.0.0.0/0"
  gateway_id             = aws_internet_gateway.this.id
}

resource "aws_route_table_association" "public" {
  for_each       = aws_subnet.public
  subnet_id      = each.value.id
  route_table_id = aws_route_table.public.id
}

# Private route tables — one per AZ so each AZ uses its own local NAT.
# This is what gives you AZ-level failover: AZ-a traffic never depends on AZ-b.
resource "aws_route_table" "private" {
  for_each = toset(local.azs)
  vpc_id   = aws_vpc.this.id
  tags     = { Name = "${local.name_prefix}-rt-private-${each.key}" }
}

resource "aws_route" "private_nat" {
  for_each               = aws_route_table.private
  route_table_id         = each.value.id
  destination_cidr_block = "0.0.0.0/0"
  # If single_nat_gateway, every AZ points at the one NAT; otherwise its own.

}

# Associate all three private tiers in each AZ with that AZ's private table.

  for_each       = aws_subnet.private_small
  subnet_id      = each.value.id
  route_table_id = aws_route_table.private[each.key].id
}

# ---------------------------------------------------------------------------
# Shared bootstrap fragment: mount the dedicated EBS data volume and discover
# cluster peers by tag. Written as $VAR (no braces) so the bash survives
# Terraform's heredoc interpolation; ${var.*} below are Terraform values.
# ---------------------------------------------------------------------------
locals {
  bootstrap_common = <<-BASH
    #!/bin/bash
    set -euxo pipefail
    dnf -y update

    # --- mount the dedicated data volume (second NVMe disk) at /data ---
    DATA_DEV=$(lsblk -dpno NAME,TYPE | awk '$2=="disk" && $1 !~ /nvme0n1/ {print $1; exit}')
    if [ -n "$DATA_DEV" ]; then
      blkid "$DATA_DEV" || mkfs -t xfs "$DATA_DEV"
      mkdir -p /data
      echo "$DATA_DEV /data xfs defaults,nofail 0 2" >> /etc/fstab
      mount -a
    fi

    # --- discover cluster peers by tag (set CLUSTER before sourcing) ---
    TOKEN=$(curl -s -X PUT http://169.254.169.254/latest/api/token -H "X-aws-ec2-metadata-token-ttl-seconds: 300")
    REGION=$(curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/placement/region)
    SELF_IP=$(curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/local-ipv4)
  BASH
}

  instance_type          = var.kafka_instance_type
  subnet_id              = local.large_subnet_ids[count.index % length(local.large_subnet_ids)]

    volume_size = 30
    volume_type = "gp3"

  ebs_block_device {
    device_name           = "/dev/sdf"
    volume_size           = var.kafka_data_gb
    volume_type           = "gp3"
    encrypted             = true
    delete_on_termination = false
  }

  user_data = <<-BASH
    ${local.bootstrap_common}
    CLUSTER="${local.name_prefix}-kafka"
    PEERS=$(aws ec2 describe-instances --region "$REGION" \
      --filters "Name=tag:Cluster,Values=$CLUSTER" "Name=instance-state-name,Values=running,pending" \
      --query 'Reservations[].Instances[].PrivateIpAddress' --output text | tr '\t' '\n' | sort)

    dnf -y install java-17-amazon-corretto-headless
    # >>> Install Kafka ${var.kafka_version} here (download the official
    #     binary tarball to /opt/kafka). Self-hosting = your choice of source.
    #
    # Write /data/kafka + a config you own. KRaft needs a quorum of voters;
    # build it from the discovered $PEERS. Example shape:
    #   process.roles=broker,controller
    #   controller.quorum.voters=<id1>@<ip1>:9093,<id2>@<ip2>:9093,...
    #   listeners=PLAINTEXT://$SELF_IP:9092,CONTROLLER://$SELF_IP:9093
    #   log.dirs=/data/kafka
    mkdir -p /data/kafka
    printf 'peers:\n%s\n' "$PEERS" > /data/kafka/CLUSTER_PEERS.txt
  BASH

  tags = {
    Name    = "${local.name_prefix}-kafka-${count.index}"
    Tier    = "private-large"
    Role    = "kafka"
    Cluster = "${local.name_prefix}-kafka"
  }
}

The same network, built by hand with the AWS CLI

Terraform is the recommended way (it remembers state and can undo cleanly). But seeing the raw AWS CLI commands makes it crystal clear what Terraform is doing for you. Below is the same build, step by step. Each command prints an ID; we capture it in a shell variable and feed it to the next command. This shows just AZ-a's subnets to stay readable — repeat the subnet/route steps for AZ-b and AZ-c.

Before you start: install the AWS CLI v2 and run aws configure with credentials that can create VPC resources. Set your region with export AWS_REGION=us-east-1.

Step 1 — Create the VPC

VPC_ID=$(aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=platform-prod-vpc}]' \
  --query 'Vpc.VpcId' --output text)

# Turn on DNS so resources can resolve names and get DNS hostnames
aws ec2 modify-vpc-attribute --vpc-id "$VPC_ID" --enable-dns-support
aws ec2 modify-vpc-attribute --vpc-id "$VPC_ID" --enable-dns-hostnames
echo "VPC: $VPC_ID"

Step 2 — Internet Gateway (the public front door)

IGW_ID=$(aws ec2 create-internet-gateway \
  --query 'InternetGateway.InternetGatewayId' --output text)
aws ec2 attach-internet-gateway --vpc-id "$VPC_ID" --internet-gateway-id "$IGW_ID"

Step 3 — The four subnets in AZ-a

# Public /24

# Auto-assign public IPs in the public subnet (this is a per-subnet setting)
aws ec2 modify-subnet-attribute --subnet-id "$PUB_A" --map-public-ip-on-launch

# Private-small /26
PS_A=$(aws ec2 create-subnet --vpc-id "$VPC_ID" --availability-zone us-east-1a \

# Private-medium /24
PM_A=$(aws ec2 create-subnet --vpc-id "$VPC_ID" --availability-zone us-east-1a \
  --cidr-block 10.0.32.0/24 \

# Private-large /22
PL_A=$(aws ec2 create-subnet --vpc-id "$VPC_ID" --availability-zone us-east-1a \

# A NAT needs a public Elastic IP and lives in a PUBLIC subnet
EIP_A=$(aws ec2 allocate-address --domain vpc --query 'AllocationId' --output text)

NAT_A=$(aws ec2 create-nat-gateway \
  --subnet-id "$PUB_A" \

# --- PUBLIC route table: send internet traffic to the Internet Gateway ---
RT_PUB=$(aws ec2 create-route-table --vpc-id "$VPC_ID" \

Section 8

Placing real workloads in this network

Now the payoff: where does each piece of a real platform actually go? Below is a tier-by-tier home for messaging (Kafka, SNS, SQS), databases (Postgres), search (OpenSearch), dataflow (NiFi), and the web/API layer (Java & Node.js). In this build the stateful stores are self-hosted on EC2 (Kafka, Postgres, OpenSearch) so you control every config file; SNS/SQS stay AWS-native and are reached privately through VPC endpoints. The rule repeats: anything stateful or sensitive goes in private subnets; only the load balancer faces the world \u2014 and only on the ranges you allow. The next section shows exactly how they all wire together.

Web & API tier — Java + Node.jsprivate-medium /24

Your Spring Boot / Express services run on EC2, ECS, or EKS in the medium subnets. They never get public IPs. Public users hit an Application Load Balancer in the public subnet on 80/443, but only from the ranges you allow-list (allowed_web_cidrs \u2014 the "limited public web"); the ALB forwards to the app SG on 8080. Rolling deploys briefly double your running tasks, so the /24 (\u2248251 usable) gives breathing room.

Why medium: stateless app servers scale moderately and recycle often; they don't need the huge address space the data tier does.

Streaming — Apache Kafka (self-hosted on EC2)private-large /22

You run the brokers yourself on EC2 \u2014 one per AZ (3 minimum for production), in KRaft mode (no ZooKeeper). Each broker gets a dedicated encrypted EBS log volume and a private IP; clients (the app tier and NiFi) open many connections; partitions replicate across AZs. Listener ports 9092\u20139098 are open only from the app and NiFi SGs and broker-to-broker (the self = true rule). As you add brokers and partitions, IP usage climbs \u2014 hence the large /22.

Why self-host: total control of server.properties, the exact Kafka version, and JVM tuning. Failover: brokers in 3 AZs with replication factor 3 keep the cluster writable if one AZ fails.

Messaging — Amazon SNS & SQSprivate-small /26 (via VPC endpoints)

SNS (publish/subscribe fan-out) and SQS (durable queues) are managed services — you don't run servers for them. But to reach them without leaving the private network, you place Interface VPC Endpoints in the small subnets. Your apps then call SQS/SNS over a private path, so message traffic never touches the public internet. The endpoints need very few IPs, so a /26 closet is perfect.

Pattern: SNS topic → fans out to multiple SQS queues → each consumed by a different service. This decouples producers from consumers and absorbs traffic spikes.

You run Postgres on EC2 in the large subnets, reachable only on 5432 from the app and NiFi SGs. Each node keeps its data directory on a dedicated encrypted EBS volume that survives instance replacement.

Why self-host: you own postgresql.conf and pg_hba.conf, choose the major version, and tune WAL/replication yourself. Why large: a primary + standbys + future growth adds up; the /22 leaves room. Egress is tight \u2014 only HTTPS out for patching.

You run OpenSearch yourself on EC2 data nodes across 3 AZs. NiFi moves and transforms data between systems (e.g., pulling from Kafka, writing to OpenSearch or Postgres). A clustered NiFi of a few nodes fits comfortably in medium. It has its own security group and is named as an allowed source on the Kafka, Postgres, and OpenSearch SGs, so it reaches the large tier over internal rules only. Its secured UI/API is on 8443, open only from inside the VPC \u2014 you reach it through SSM Session Manager port-forwarding or an internal load balancer, never publicly.

Cluster nodes coordinate on 11443 and load-balance on 6342 (both self). If your NiFi cluster is very large or ingests massive volumes, promote it to the large tier.

Section 9

Self-hosted data services & how they wire together

Here's the whole reason for the network. A limited public web reaches a single load balancer, which forwards to an API tier, which is the only thing allowed to talk to the backend data services. Those services — Kafka, Postgres, OpenSearch, and NiFi — are self-hosted on EC2 so you own every config file, while SNS and SQS are AWS-native and reached privately through VPC endpoints. Nothing that holds data is ever directly exposed to the internet.

The traffic path, end to end

1. A request arrives from an allow-listed range and hits the ALB in the public subnets on 80/443. Everyone else is blocked at the security group.

2. The ALB forwards to the app/API instances on 8080 — and the app SG only accepts traffic from the ALB SG, so you can't bypass the balancer.

3. The API queries the backends on their own ports: Kafka (9092–9098), Postgres (5432), OpenSearch (9200), and publishes/polls SNS & SQS over the private VPC endpoints (443).

4. NiFi runs the same data plane independently — pulling from and pushing to the same stores over internal SG rules.

5. The stores cluster privately among themselves: Kafka inter-broker, Postgres streaming replication, OpenSearch 9300 transport, NiFi cluster ports — all via self rules, never leaving the VPC.

Why self-host every data store?

You asked for full control of configuration, and that's the core trade-off below. Self-hosting buys you total control and portability; managed services buy you back your operations time. SNS and SQS are the exception — there is no self-hosted version, so we consume them as AWS services over private endpoints.

Reaching the services that have no public door

NiFi's UI and OpenSearch deliberately have no internet route. To administer them you don't open a hole — you tunnel in. SSM Session Manager port-forwarding gives you a local port that maps straight to a private instance, with no bastion, no inbound SSH, and full IAM audit logging:

aws ssm start-session \
  --target i-0nifi... \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["8443"],"localPortNumber":["8443"]}'

Then open https://localhost:8443. For shared team access, put an internal ALB/NLB in front of NiFi or OpenSearch instead — still entirely inside the VPC.

Section 10

Real-world use cases, pros & cons

A company that needs different levels of security internally and externally is exactly who this design serves. Here are three common shapes, with honest trade-offs.

Use case A — Customer-facing SaaS with sensitive data

Public web app, private data. Externally only the ALB on 443 is reachable; internally, tiers are isolated so a compromised web server still can't reach the database directly (it must come through the app SG on 5432).

Pros
  • + Tiny external attack surface.
  • + Clear blast-radius limits between tiers.
  • + Easy to pass audits (PCI/SOC 2) thanks to segmentation + Flow Logs.
Cons
  • − More moving parts than one flat subnet.
  • − NAT + endpoints add cost.
  • − Cross-AZ data transfer charges apply.

Use case B — Internal-only enterprise platform (no public access)

No Internet Gateway at all for app traffic; users arrive over a VPN or AWS Direct Connect from the corporate network. Everything lives in private subnets. The "external" boundary is your corporate edge, not the open internet.

Pros
  • + Strongest posture — nothing is internet-reachable.
  • + Natural fit for regulated industries.
  • + Simple mental model: all private.
Cons
  • − Requires VPN/Direct Connect plumbing.
  • − Patching/updates rely on NAT or endpoints.
  • − Harder for third parties to integrate.

Use case C — Event-driven data platform (Kafka + SNS/SQS + OpenSearch)

High-throughput ingestion: producers publish to Kafka/SNS, consumers process via SQS, results land in OpenSearch and Postgres, NiFi orchestrates flows. The large tier does the heavy lifting; messaging stays private via endpoints.

Pros
  • + Decoupled, spike-tolerant (queues buffer load).
  • + Scales each tier independently.
  • + Large subnets absorb cluster growth.
Cons
  • − Most complex to operate & reason about.
  • − Many Interface endpoints add hourly cost.
  • − Cross-AZ replication increases data charges.
The honest summary: tiered, multi-AZ, private-by-default networking costs more (NAT gateways, endpoints, cross-AZ transfer) and has more parts than a single flat subnet. You buy real security isolation and resilience with that complexity. For a throwaway demo, a default VPC is fine. For anything holding customer data or needing uptime, this structure pays for itself the first time it contains an incident or survives an AZ outage.

Section 11

Failover & operational controls

"Failover" means: when one part breaks, another quietly takes over so users barely notice. Our design builds this in at the network layer; you then layer service-level redundancy on top.

Multi-AZ everything

Every tier exists in all three AZs. The ALB health-checks targets and stops routing to a dead AZ automatically. Your Postgres standbys (streaming replication) can be promoted to primary if the primary's AZ fails. Kafka replication keeps data available. OpenSearch rebalances shards. One datacenter can fail and the system keeps serving \u2014 though with self-hosting, you own the promotion/failover automation.

NAT per AZ vs single NAT — the explicit trade-off

Our single_nat_gateway variable controls this. One NAT per AZ (prod default) means AZ-a's outbound traffic never depends on AZ-b — true isolation, at roughly one NAT's hourly + data cost per AZ. One shared NAT (dev) is cheaper but if its AZ fails, every private subnet loses outbound internet. Flip the flag per environment.

Per-AZ private route tables

We give each AZ its own private route table pointing at that AZ's local NAT. This is the subtle detail that makes NAT-per-AZ actually isolate failures instead of silently cross-wiring AZs.

Visibility & guardrails

  • Flow Logs → audit trail of every accepted/rejected connection.
  • Default tags → every resource tagged with environment/project/ManagedBy for cost tracking and ownership.
  • Pinned provider → reproducible applies, no surprise upgrades.
  • Remote state + locking (commented in versions.tf) → safe team collaboration.
  • CIDR validation in variables → catch typos at plan time, not after a broken apply.

Where to grow next

Natural follow-ons that fit this foundation: AWS WAF on the ALB for L7 protection, AWS Network Firewall for deep packet inspection, Transit Gateway to connect multiple VPCs/accounts, PrivateLink to expose internal services privately to partners, and GuardDuty for threat detection reading those Flow Logs.
